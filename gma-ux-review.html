Solid analytical foundation — refinements needed on density and table ergonomics

The GMA dashboard handles a genuinely complex data model well. The core design decisions are sound: a light theme works for this analyst context, the color semantic system (debit/credit with directional arrows) is correct, drill-through navigation is a meaningful productivity feature, and the exchange-rate disclosure shows regulatory awareness. The issues holding this back from an A are concentrated in three areas: the Transfers tab table is too wide to read comfortably at any practical screen size; the tab strip has a navigation ordering problem that buries the most-used tab (Transactions); and the filter bar on the Transactions tab lacks an active-state indicator, making it impossible to know at a glance what filters are applied. These are all fixable without a design overhaul.

Critical Findings — Fix Before Release
C-01 Critical Transfers Tab: 13–15 Column Table Is Illegible at Any Practical Viewport

What is broken: The Intra-Entity table has 13 columns and the Intercompany table has 15 columns. At 1400px (the container max-width), the 13-column version has roughly 107px per column on average — but several columns compete: two bank name columns need at least 180px each, two date-pair columns each need 80px, account number badges want 45px each, and the description column wants 200px+. The math does not work. In practice the table overflows horizontally, requires scrolling, and the four date columns (Posted Out, Posted In, Value Out, Value In) read as an undifferentiated wall of dates because they have no visual grouping to show which pair belongs to the same side of the transfer.

Root cause: Four date columns were added to show float calculation transparency, which is a good analytical decision — but the column design was not reconsidered when they were added. The account number columns (Acct at 45px) also show only the last 4 digits with no label indicating this is truncated, which is a separate data trust issue.

Recommendation — Grouped Column Headers: Use a two-row header structure. Group the four date/float columns under two parent headers: "Posted Date" (spanning Out, In, Float) and "Value Date" (spanning Out, In, Float). This adds visual parsing structure and allows the parent header to carry the explanatory label so the child headers can be abbreviated to "Out", "In", "Float". Also merge the two bank + account columns into a single compound cell showing bank name with the 4-digit account suffix below it in smaller monospace text. This reduces 4 columns to 2, cutting the total from 13 to 11 for intra-entity and from 15 to 12 for intercompany.

C-02 Critical No Active Filter Indicator — Users Cannot Tell What Is Filtered

What is broken: The Transactions tab has 7 active filter controls. When a user drills through from a summary table (e.g., clicking a bank row in the Banks tab), the dashboard jumps to the Transactions tab with that filter pre-applied. But the filter bar looks identical whether 0 or 5 filters are active — every dropdown stays the same visual weight. A user who returns to the Transactions tab from another tab has no indication that the view is showing a subset of data, not the full dataset. This is a data trust problem: the user may draw incorrect conclusions from a filtered subset they've forgotten is filtered.

Recommendation: Add an active-filter banner that appears below the filter bar whenever any filter is non-default. It should list what is active and include a "Clear All" link. Additionally, update the Transactions tab button label dynamically to show the filtered count (e.g., "Transactions (342)" vs "Transactions").

/* Add after the filter bar, before the table-wrap */
<div id="filterBanner" style="display:none; background:#eff6ff; border-bottom:1px solid #bfdbfe;
  padding:8px 20px; font-size:12px; color:#1e40af; display:flex;
  justify-content:space-between; align-items:center;">
  <span id="filterBannerText"></span>
  <button onclick="clearFilters()" style="font-size:12px; color:#1e40af;
    background:none; border:none; cursor:pointer; text-decoration:underline;">
    Clear all filters
  </button>
</div>

/* In renderTxns(), update the banner */
function updateFilterBanner() {
  const parts = [];
  if (countrySel.value) parts.push('Country: ' + countrySel.value);
  if (bankSel.value)    parts.push('Bank: ' + (BANK_DISPLAY[bankSel.value]||bankSel.value));
  if (catSel.value)     parts.push('Type: ' + catSel.value);
  if (flowSel.value)    parts.push('Flow: ' + flowSel.value);
  const banner = document.getElementById('filterBanner');
  if (parts.length) {
    document.getElementById('filterBannerText').textContent
      = 'Active filters: ' + parts.join(' | ');
    banner.style.display = 'flex';
  } else {
    banner.style.display = 'none';
  }
}
C-03 Critical Tab Navigation Order Buries the Most-Used Tab at Position 10

What is broken: The current tab order is: Overview, Cash Flow, Country, Transaction Types, Banks, Monthly Trends, Country Review, Transfers, Insights, Transactions. The Transactions tab — the primary data explorer where analysts spend the most time — is the last item in a 10-tab strip. Hick's Law applies directly here: more choices + wrong ordering = slower time-to-action for the primary task. Country Review is also positioned before Transfers despite being a secondary analytical layer. The "Country" and "Country Review" tabs having similar names 3 positions apart also creates confusion about what each does.

Recommendation — Revised tab order based on task frequency:

PositionTab LabelRationale
1OverviewLanding — KPI summary, entry charts
2TransactionsMove to second — primary exploration tab
3TransfersHigh-value analyst task — float analysis
4BanksDrill-down anchor — used frequently
5Cash FlowFlow type analysis
6Transaction TypesCategory breakdown
7Monthly TrendsTemporal analysis
8CountryMerge "Country" + "Country Review" into one tab with sub-sections
9InsightsFindings + recommendations — natural endpoint

Merging "Country" and "Country Review" into a single "Country" tab eliminates the duplicate-name confusion. The tab can open showing the summary cards/charts at the top, with the detailed per-country breakdowns below — already the natural scroll order since country_review_html generates the detail sections.

High Priority Findings
H-01 High Transfer Table Row Height and Font Size Are Below Minimum Legibility Threshold

What is broken: The four date columns in the Transfers table use inline font-size:11px and the PF/VF float badge columns use font-size:11px with no minimum row padding. At 11px, WCAG 1.4.4 (Resize Text) requires the user be able to scale to 200% without loss of function, but beyond that, 11px at normal zoom on a standard 96 DPI screen is genuinely uncomfortable for sustained reading — especially for date strings which are already dense character sequences. The PF/VF column headers are also cryptic abbreviations with only a title attribute tooltip for explanation, which is inaccessible to keyboard and touch users.

Recommendation: Set a minimum font size of 12px for all table data cells. Replace PF/VF header abbreviations with "Float (d)" label — short enough to fit but self-explanatory. If the grouped header approach from C-01 is implemented, the parent header "Posted Date" provides enough context that the child can say "Float" without needing expansion.

Current row padding is padding: 8px 12px on td globally. The Transfers table should add a specific override: #tfTable td { padding: 9px 12px; } — adding 1px vertical padding reduces the cramped feeling without meaningfully increasing table height.

H-02 High Inline Hardcoded Status Colors Bypass the Token System and Fail AA at Small Sizes

What is broken: Several components use hardcoded hex values for status colors rather than CSS custom properties. The Priority Action Matrix in the Insights tab uses inline style="color:#c53030", color:#b7791f, and color:#718096 directly on <td> elements at 13px body size — but more problematically, #b7791f at 13px on white achieves only 3.9:1 contrast ratio, which fails WCAG AA (requires 4.5:1 for normal text). The Bank Consolidation Candidates table uses the same three-color coding with the same failure on the warning amber.

The KPI float value coloring at lines 1085–1086 also uses hardcoded hex triples inline on style attributes: '#276749' if avg_float < 1 else '#b7791f' if avg_float < 3 else '#c53030'. This pattern appears four times in the Transfers KPI row. Any future color system change requires hunting through Python f-strings rather than updating one CSS token.

Recommendation: Add three semantic tokens to :root:

:root {
  /* existing tokens ... */
  --status-ok:   #1a6640;   /* reuse --green — 6.9:1 */
  --status-warn: #92400e;   /* darker amber — 7.2:1 — replaces #b7791f */
  --status-crit: #c53030;   /* reuse --accent — 5.5:1 */
}

Then in Python, replace the f-string color ternary with CSS class names:

# Before (hardcoded hex in f-string):
f"color:{'#276749' if avg_float < 1 else '#b7791f' if avg_float < 3 else '#c53030'}"

# After (class-based):
float_cls = 'status-ok' if avg_float < 1 else 'status-warn' if avg_float < 3 else 'status-crit'
# in template:
f'class="val {float_cls}"'
H-03 High Transactions Table Pagination: Count Indicator Position and "Rows" Selector Ordering

What is broken: The pagination bar layout is: [Showing X-Y of Z] [Rows: select] [Previous] [Next]. There are two problems. First, the rows-per-page selector sits between the count display and the navigation buttons — visually it breaks the reading flow from "here is where you are" to "here is how to navigate". Standard treasury table pagination (Bloomberg, Mercury, Stripe) positions the count on the left and navigation controls on the right as a unit. Second, the count reads "Showing 1-200 of 39,856" but does not update the label to reflect that the 200-row page size is currently active — if a user changes the page size, the count updates numerically but there is no summary like "200 rows per page".

Recommendation: Reorder the pagination bar to: [left: count + rows-per-page label] [right: Previous / Next buttons]. This is a CSS flex change:

/* Updated pagination layout */
.pagination {
  display: flex;
  justify-content: space-between;  /* left group vs right group */
  align-items: center;
  padding: 10px 20px;
  background: #f8fafc;
  border-top: 1px solid var(--border);
  font-size: 13px;
}
/* Left group: count + page size */
.pagination-left  { display: flex; align-items: center; gap: 16px; }
/* Right group: prev/next */
.pagination-right { display: flex; align-items: center; gap: 8px; }
H-04 High Tab Strip Wraps Onto Two Lines at 1024px — Creates Visual Confusion

What is broken: The tab strip uses flex-wrap: wrap which means at narrower viewports — specifically around 1024–1200px where many analyst laptops operate — the 10 tabs wrap onto a second line. The active tab underline indicator (border-bottom: 2px solid var(--primary); margin-bottom: -2px) is designed to sit on the container border, but when tabs wrap, the second-line tabs are no longer aligned to that border, creating a visual disconnect where the active indicator floats mid-container.

Recommendation: Two viable approaches depending on whether C-03 tab consolidation is implemented. If reducing to 9 tabs as recommended in C-03, add overflow-x: auto instead of flex-wrap: wrap. This keeps tabs on a single row and adds horizontal scrolling at narrow viewports — much cleaner than wrapping. If keeping 10 tabs, reduce tab padding from 10px 20px to 9px 14px and use font-size: 13px — the 10 tabs fit on a single row down to 1024px.

.tabs {
  display: flex;
  gap: 0;
  border-bottom: 2px solid var(--border);
  margin-bottom: 20px;
  overflow-x: auto;          /* replaces flex-wrap: wrap */
  -webkit-overflow-scrolling: touch;
  scrollbar-width: thin;
  scrollbar-color: var(--border) transparent;
}
.tab {
  padding: 9px 14px;          /* reduced from 10px 20px */
  font-size: 13px;             /* reduced from 14px */
  white-space: nowrap;         /* prevent individual tab text wrapping */
  flex-shrink: 0;              /* prevent tabs from shrinking below their text width */
}
H-05 High Transfer Table Pagination Does Not Reset on Filter Change

What is broken: The renderIntra() function is called from the filter oninput and onchange handlers, but intraPage is not reset to 0 inside those handlers. It is only reset inside renderIntra() itself when the "Clear" button is pressed. This means if a user is on page 3 of 200 transfers and then types a search term that matches only 15 transfers (fewer than one page), the pagination can report "Showing 201–200 of 15" — an impossible range — before rendering the empty state. The same bug exists in renderIC().

Recommendation: Add page reset at the start of the filter change event:

// Intra date inputs and search — add intraPage = 0 reset
<input ... onchange="intraPage=0; renderIntra()">
<input ... oninput="clearTimeout(intraTimer); intraTimer=setTimeout(() => { intraPage=0; renderIntra(); }, 300)">

// Same fix for IC table
<input ... onchange="icPage=0; renderIC()">
Medium Priority Findings
M-01 Medium Overview Tab: Chart Pair Sizing Imbalance — Monthly Chart Visually Dominates

The Overview tab renders a 2-column chart row (category bar + bank bar) then a full-width monthly chart. The monthly chart with 3 datasets (debit bars, credit bars, net line) at max-height: 350px full-width feels proportionally heavy against the two shorter charts above. The visual hierarchy implies monthly trends are the primary insight, which may not always be true.

Recommendation: Cap the monthly chart at 280px max-height (reducing from 350px) and add a summary stat bar above it with the largest month by net flow and the month-over-month trend direction. This gives the chart more context without requiring users to hover to see what the biggest month was. Alternatively, restructure the Overview to show three full-width sections: (1) KPI cards (already present, moved above tabs), (2) the two-column chart pair, (3) an insight summary strip that auto-generates from Python rather than a large chart.

M-02 Medium Cash Flow Tab KPI Cards Use Raw USD Without M/K Abbreviation — Numbers Overflow

The Cash Flow tab KPI cards display values like +$12,450,832 using the format ${collection_credits_usd:,.0f}. At a KPI card width of ~220px with font-size clamp at 16–24px, a 13-character dollar string will overflow the white-space: nowrap container at smaller card sizes. The top-level KPI cards on the page correctly use /1e6:,.1f M-suffix formatting — but the Cash Flow tab KPIs do not. This inconsistency also creates a cognitive mismatch: the header says "$12.5M" but the Cash Flow tab says "$12,450,832" for the same number.

Recommendation: Apply the same M-suffix format used in the top-level KPIs throughout all KPI cards. Specifically update the Cash Flow tab KPI values to use the /1e6:,.1f}M Python format string, and add a title attribute or secondary .sub line with the full precision value for users who need it.

M-03 Medium Drill-Through Navigation Does Not Provide a "Back" Affordance

Clicking a row in any summary table (Banks, Categories, Country, Flow) jumps the user to the Transactions tab with a pre-applied filter. This is a useful feature, but it has no "back" affordance. After landing on the filtered Transactions view, the user has no indication that they navigated from another tab, and no quick way to return to the source table to pick a different row. This forces users to manually click back to the source tab and re-orient in a potentially long table.

Recommendation: After a drill-through navigation, show a breadcrumb banner at the top of the Transactions tab content (above the filter bar): "Viewing transactions for: Citibank Ivory Coast  |  Back to Banks". This can be implemented by storing the source tab ID in a variable during the drill-through click handler and rendering it as part of the filter banner from C-02.

M-04 Medium Category Bar Chart: X-Axis Labels Rotate 45° and Still Truncate

The category chart (catChart) uses maxRotation: 45 and font: { size: 10 } on the x-axis ticks. Category names like "SWIFT / Telex", "Check / Cheque", and "Fees / Commissions" are 15–20 characters — at 10px with 45° rotation they are still truncated in Chart.js with an ellipsis. Analysts cannot read which category a bar represents without hovering, which defeats the chart's purpose as a quick comparison view.

Recommendation: Convert the category chart from vertical bars to a horizontal bar chart. Set indexAxis: 'y', keep font: { size: 11 } on y-axis labels, and set a fixed height of 320px. The bank chart already correctly uses horizontal orientation — apply the same logic to category. This is a one-word change in the Chart.js config.

new Chart(document.getElementById('catChart'), {
  type: 'bar',
  data: { ... },
  options: {
    indexAxis: 'y',      /* flip to horizontal */
    responsive: true,
    plugins: { legend: { display: false } },
    scales: {
      x: { ticks: { callback: v => v.toLocaleString() } },
      y: { ticks: { font: { size: 11 } } }  /* labels readable at 11px horizontal */
    }
  }
});
M-05 Medium Country Pie Chart Uses Semantically Charged Colors for Non-Semantic Data

The country pie chart uses the color array ['#3182ce','#c53030','#276749','#b7791f']. The second color is #c53030 — the same red used throughout the dashboard to mean "debit/outflow". In the pie chart context, it is assigned to whichever country sorts second alphabetically. If an analyst's eye lands on the red slice and associates it with "negative/debit", they will misread the chart — this is a direct application of the established Stroop-effect interference between learned semantic colors and new non-semantic uses of the same hue.

Recommendation: Use a neutral categorical palette for the country pie. Blue family only, or blue+teal with no red/green:

// Country pie — neutral categorical palette (no red/green)
backgroundColor: ['#3182ce', '#0891b2', '#7c3aed', '#0f766e']
// Blues/teals/violet — clearly non-semantic, still readable

Apply the same logic to the country bar chart, which currently uses rgba(197,48,48,0.7) for debits and rgba(39,103,73,0.7) for credits — which is actually correct and semantically appropriate for a debit/credit comparison chart. Only the pie chart (which is not a debit/credit chart) needs the fix.

M-06 Medium Accounts Tab Is Missing — Referenced in Brief but Not in Dashboard

The review brief lists 10 tabs including an "Accounts" tab for account-level detail. This tab does not appear in the build_html.py source — the tab strip generates 10 tabs (Overview, Cash Flow, Country, Transaction Types, Banks, Monthly Trends, Country Review, Transfers, Insights, Transactions), none of which is "Accounts". The Banks tab shows bank-level aggregates, not individual account-level data. If the dataset contains account-level identifiers (the Transfers table shows source_account and dest_account fields with last-4 masking), an account-level view would be analytically valuable — showing per-account transaction count, balance direction, and float behaviour.

Recommendation: If account-level data is available in ctm_data.json, build an Accounts tab similar in structure to the Banks tab but grouped by account number (masked to last-4 for display). This is particularly useful for treasury teams managing multiple accounts per bank. If account data is not present in the dataset, this is a data gap to note in the Insights tab rather than a dashboard gap.

Low Priority / Polish Findings
L-01 Low Keyboard Navigation: Tab Buttons Lack aria-controls Association

The tab buttons have role="tab" and aria-selected toggling (correctly implemented in showTab()), but are missing aria-controls="[panel-id]" and the corresponding aria-labelledby on each panel. The role="tabpanel" is present on panels. Without aria-controls, screen reader users navigating by ARIA landmarks cannot directly jump from the tab to its panel. This is a partial ARIA pattern implementation — the skeleton is correct but the association attributes are missing.

<!-- Tab button -->
<button class="tab" role="tab"
  id="tab-overview"
  aria-controls="overview"
  aria-selected="true"
  onclick="showTab('overview')">Overview</button>

<!-- Tab panel -->
<div id="overview" class="tab-content active" role="tabpanel"
  aria-labelledby="tab-overview">...</div>
L-02 Low Insights Tab: Priority Action Matrix Number Column Uses Color-Only Differentiation

The Priority Action Matrix uses color:#c53030 for priorities 1–2, color:#b7791f for 3–5, and default text for 6–8. There is no other visual differentiator — no icon, no badge, no label. A user who cannot distinguish red from orange (deuteranopia affects ~5% of male users) will see all priorities in the same visual weight. Additionally, the priority numbers are shown without any label — "1" in red doesn't communicate "Critical" to a first-time viewer without a legend.

Recommendation: Replace the raw number with a priority badge: <span class="pill pill-critical">P1</span>, <span class="pill pill-high">P3</span> etc. This reuses the existing pill classes from the UX report pattern and adds text labels that are readable independent of color.

L-03 Low Transfer Table "Acct" Column Header Does Not Indicate Masking

The account number column header is labeled "Acct" and shows only the last 4 digits of the account number (via .slice(-4)). There is no indication to the user that the value is truncated. A user unfamiliar with the system might try to use the 4-digit value to identify an account in their bank portal and be confused. The full account is presumably available in the underlying data — the masking is a display choice for security/privacy.

Recommendation: Either (a) label the column header "Acct (last 4)" to make the truncation explicit, or (b) as part of the compound cell approach from C-01, render the account suffix with a masking prefix: "•••• 4521" which is the standard bank statement pattern for masked account numbers. Option (b) is preferred as it is the familiar pattern from bank portals, credit card statements, and Stripe/Plaid interfaces.

L-04 Low Missing prefers-reduced-motion Guard on CSS Transitions

The dashboard has minimal animation — just transition: all 0.15s on tab hover and button hover states — so the risk is low. However, WCAG 2.3.3 (Level AAA, Advisory) recommends honoring prefers-reduced-motion. For a dashboard that may be used in an enterprise environment where some users have vestibular disorders, this is a low-cost addition.

@media (prefers-reduced-motion: reduce) {
  .tab, .btn, .drill-row { transition: none; }
}
L-05 Low Debit/Credit Arrow Pseudo-Elements May Not Render in All Export Contexts

The .debit::before { content: '\2193 '; } and .credit::before { content: '\2191 '; } CSS rules add directional arrows to all debit and credit values. This is a good WCAG 1.4.1 non-color cue. However, when the table is exported to CSV via exportCSV(), the arrows are not included — the CSV uses the raw amount_xof numeric values, which is correct. The issue is the converse: if a user copies table cells from the DOM (keyboard select + copy), some browsers will include the CSS pseudo-element content in the clipboard, producing values like "↓ -1,250,000" in a spreadsheet paste. This is a minor annoyance but worth noting. Confirm that the sign convention (negative = debit) is consistent in the CSV export, which it is — no fix required beyond awareness.

What Is Working Well
Exchange Rate Disclosure — Proactive Trust Signal
The .rate-notice amber banner at the top of every page view correctly surfaces the fixed-rate caveat before the user sees any numbers. The note that XOF is EUR-pegged and that rates varied over the period is the right level of specificity for a treasury analyst audience. Most tools bury this footnote; surfacing it prominently is good practice.
Drill-Through Navigation Pattern
Clicking any summary row (bank, category, flow, country) pre-applies that filter and jumps to the Transactions tab. The data-filter-type / data-filter-value data attribute pattern is clean and extensible. The cascading Country → Bank filter reset on country change is correctly handled in onCountryChange().
Debit/Credit Semantic Color + Arrow Pairing
Using both color and directional Unicode arrows (↑/↓) for debit/credit distinction satisfies WCAG 1.4.1 (Use of Color, Level A). The specific choice of navy+red+dark-green achieves AA contrast on white. The fmt_net() function correctly adds +/- prefix text as a third redundant cue.
Float Badge Color Semantics Are Correct
The floatBadge() function correctly maps 0 days = green, 1–2 days = amber, 3+ days = red. This mirrors the mental model a treasury analyst has about settlement float — same-day is ideal, 1–2 days is normal, 3+ days warrants investigation. The color thresholds are calibrated to the domain, not generic traffic-light defaults.
Bank Display Name Normalization
The BANK_DISPLAY mapping converts internal system codes ("AFG BK", "SOCGEN IVORY COAST") to readable institutional names. This is applied consistently in dropdowns, table cells, and chart labels via the bdn() function. It also handles the BANK_DISPLAY[b] || b fallback pattern correctly for unmapped codes.
Cascading Country → Bank Dropdown Filter
Selecting a country in the filter bar narrows the bank dropdown to only banks operating in that country — exactly the pattern recommended in prior CTM reviews. The implementation correctly preserves the previously selected bank value if it is still valid after the country change.
Operational vs. Total Flow Separation
The Cash Flow tab correctly separates total flow (including inter-account transfers) from operational flow (excluding transfers). The two footer rows in the flow table distinguish these clearly. This prevents the misleading inflation of gross debit figures by internal pool movement — a common error in treasury dashboards.
Insights Tab Auto-Generated Narrative
The country-specific insights paragraphs (banking fragmentation, MTN mobile context, cheque volume, BICIS concentration risk) add genuine analytical value beyond raw data tables. Generating these from Python using computed thresholds (cheque >2%, opacity >15%, top-bank concentration) is a scalable pattern — it works for both GMA and CTM modes without manual editing.

Designer's Note

This is a genuinely well-engineered analytical tool for a complex domain. The GMA/CTM dual-mode flag pattern, the float analysis in the Transfers tab, and the auto-generated country narrative in Insights are the kind of design decisions that show deep domain understanding — not just layout skills. The issues I've flagged are refinements, not structural problems.

The highest-value work is in the Transfers tab. That tab contains the most analytically unique data in the dashboard (matched-pair float analysis with both posted and value dates), and it is currently the hardest to read. Implementing the two-row grouped header (C-01) and adding the active filter banner (C-02) would unlock the full value of that data for analysts who currently have to scroll horizontally to understand what they're looking at.

The tab ordering change (C-03) is a three-line edit that will measurably reduce the number of clicks to reach the primary workflow. It is the most asymmetric effort-to-impact ratio of anything in this review. Do that one first.
